1
The goal

By the end of this lab, you'll have a fully serverless system where:

Upload → Lambda fires automatically → resized version appears.

Requirements:

  • One S3 bucket with a uploads/ prefix and a thumbs/ prefix
  • A Lambda function that runs whenever a new image lands in uploads/
  • The function resizes the image to 200px wide (keep aspect ratio) and writes it to thumbs/
  • The whole thing must work end-to-end with zero EC2, zero servers, zero SSH
2
Phase 1 — Create the S3 bucket

Create a brand new S3 bucket. Suggested name: cloudwithshad-resizer-<your-initials>-<random-number> (bucket names must be globally unique).

1
Bucket name: globally unique. Use lowercase only.
2
Region: same as your other resources (consistency).
3
Block all public access: keep this ON. The Lambda will access it, not the public.
4
Versioning: off (for this lab).
5
After creation, open the bucket and create two "folders": uploads/ and thumbs/
Hint: how to create a "folder" in S3

S3 doesn't really have folders — it has object key prefixes. But the console lets you click "Create folder" and it adds a zero-byte object ending in /. That's enough to act like a folder for our trigger.

3
Phase 2 — Create an IAM role for Lambda

Your Lambda function needs permission to do three things: read from S3, write to S3, write logs to CloudWatch.

  1. Go to IAM → Roles → Create role
  2. Trusted entity type: AWS service
  3. Use case: Lambda
  4. Attach these managed policies:
    • AWSLambdaBasicExecutionRole — for CloudWatch logs
    • AmazonS3FullAccess — for read/write to S3 (we'll tighten this later)
  5. Role name: cloudwithshad-resizer-role
Exam tip

In production, you'd never use S3FullAccess — you'd write a custom policy granting access only to that ONE bucket. This is the principle of least privilege. For learning, broad permissions are fine.

4
Phase 3 — Create the Lambda function
  1. Go to Lambda → Create function
  2. Author from scratch
  3. Function name: resize-image
  4. Runtime: Python 3.12
  5. Architecture: x86_64
  6. Execution role: Use existing → cloudwithshad-resizer-role
  7. Click Create function

Once created, scroll to the Code tab. Paste this starter code:

# Lambda handler — resizes any image dropped into uploads/ import boto3 import os from PIL import Image from io import BytesIO s3 = boto3.client('s3') def lambda_handler(event, context): bucket = event['Records'][0]['s3']['bucket']['name'] key = event['Records'][0]['s3']['object']['key'] # Ignore re-triggers from our own thumb writes if not key.startswith('uploads/'): return {'status': 'skip'} # Download original obj = s3.get_object(Bucket=bucket, Key=key) img = Image.open(obj['Body']) # Resize to 200px wide, keep aspect ratio w_pct = 200 / float(img.size[0]) h_size = int(float(img.size[1]) * w_pct) img = img.resize((200, h_size), Image.LANCZOS) # Save into buffer + upload to thumbs/ buffer = BytesIO() img.save(buffer, format='JPEG') buffer.seek(0) thumb_key = key.replace('uploads/', 'thumbs/') s3.put_object(Bucket=bucket, Key=thumb_key, Body=buffer, ContentType='image/jpeg') return {'status': 'resized', 'thumb': thumb_key}

Click Deploy.

Pillow isn't in Lambda by default

The code uses the Pillow library, but Lambda's Python runtime doesn't include it. You need to attach a Lambda Layer that provides Pillow. AWS publishes one publicly. Add it under Layers → Add a layer → Specify ARN — search for the public Pillow layer for your region.

Hint: easiest way to get Pillow into Lambda

Two options:

Option A: Use Klayers' public Pillow ARN. Search "Klayers Pillow Python 3.12" + your region and you'll find the ARN. Paste it in the "Specify ARN" field.

Option B: Create your own layer. Zip a Python package with Pillow inside, upload it. More work, more learning.

For this lab, Option A is fine.

Don't forget

Set the Lambda timeout to at least 30 seconds and memory to 512 MB (Configuration → General configuration → Edit). Image resizing can take a few seconds, and the default 3-second timeout will fail.

5
Phase 4 — Wire up the S3 trigger

Now we connect S3 to Lambda. In the Lambda function page:

  1. Click "+ Add trigger" (in the function overview diagram)
  2. Select S3 from the dropdown
  3. Bucket: your resizer bucket
  4. Event types: All object create events
  5. Prefix: uploads/ ← critical — limits the trigger to the uploads folder only
  6. Tick "I acknowledge that using the same S3 bucket for both input and output is not recommended..." (we handle this in code with the prefix check)
  7. Click Add
Why the prefix matters

Without the uploads/ prefix, the trigger would fire when Lambda writes the thumb back into S3 — which would trigger another resize — infinite loop. The prefix + the early-return in our code both prevent this.

6
Phase 5 — Test it!

The moment of truth.

  1. Open your S3 bucket in the console
  2. Navigate into uploads/
  3. Click Upload and drop in any JPG or PNG (a selfie, a screenshot, anything under 5 MB)
  4. Wait ~5 seconds
  5. Go to the thumbs/ folder and refresh
  6. You should see your image — resized to 200px wide 🎉

If you check the Lambda function's Monitor tab, you'll see:

  • Invocation count: 1
  • Average duration: 1-3 seconds
  • Errors: 0 (hopefully)
Submission

Hint: what to do when it doesn't work

Go to CloudWatch → Log groups → /aws/lambda/resize-image. Open the most recent log stream. The error will be in plain English. Most common issues:

  • No module named 'PIL' → Pillow layer not attached
  • Task timed out → bump timeout from 3s to 30s
  • AccessDenied → IAM role missing S3 permissions
  • Nothing in logs at all → trigger isn't firing. Check prefix & bucket name.

🚀 Bonus challenges (for the keen)

Already done with the main lab? Try one of these — each one teaches a new pattern:

  • Multiple sizes: Also create thumbs-small/ (100px) and thumbs-large/ (500px) versions in one Lambda run.
  • Format conversion: Convert any input to WebP for better compression — and append .webp to the key.
  • Watermark: Use Pillow's ImageDraw to stamp "cloudwithshad" in the bottom-right corner of every thumb.
  • Notification: Add SNS — send yourself an email every time a new thumb is created.
7
Clean up

Lambda + S3 won't break the bank, but tidy habits matter. After your screenshot:

  1. Empty the S3 bucket first (Bucket → Empty), then Delete the bucket
  2. Delete the Lambda function (Lambda → resize-image → Delete)
  3. Delete the IAM role (IAM → Roles → cloudwithshad-resizer-role → Delete)
  4. Optional: delete the CloudWatch log group (/aws/lambda/resize-image) — logs are charged after 5 GB
The honest truth

Serverless changes how you think. With EC2 you ask "what server do I need?" With Lambda you ask "what should trigger when?" Both have their place — but the future is event-driven. You just built one. Welcome to serverless.
